# v2.6.0 ASI Storm Operations and Backpressure

v2.6.0 introduces ASI Storm terminology as a TDS design concept, not as an external industry standard.

An **ASI Storm** is a transient period of extreme computational pressure generated by autonomous or multi-agent AI systems. A small number of external requests may amplify into many internal chunk, index, radix, Swiss-table, scheduling, native execution, and telemetry operations.

## Design rule

TDS should prefer being temporarily less informative over becoming unstable.

Under pressure, the telemetry layer degrades first. Chunk admission, index integrity, radix routing, and native-engine correctness remain higher priority than dashboard detail.

## One-way telemetry pipeline

The browser console consumes immutable snapshots only:

```text
native engine -> non-blocking counters/events -> telemetry publisher -> immutable snapshot -> browser panels
```

The browser does not hold engine locks, does not walk live radix or Swiss-table structures, and does not call into the hot path for refresh work.

## Pressure modes

```text
NORMAL
WARM_PRESSURE
HIGH_PRESSURE
CRITICAL_PRESSURE
PROTECTIVE_READ_ONLY
RECOVERY
```

These modes are exposed in `snapshot["pressure"]` along with a bounded pressure score.

## Semantic VFS states

```text
CREATED
ACTIVE
PRESSURIZED
THROTTLED
READ_ONLY
RECOVERING
RETIRED
```

The VFS state is derived from the pressure mode and is intended for dashboard and operator feedback.

## Chunk lifecycle states

```text
ALLOCATED -> WRITING -> SEALED -> VERIFIED -> INDEXED -> EXPOSED
                                      \-> QUARANTINED
```

TDS v2.6 surfaces lifecycle counters so ASI-storm behavior can be observed without coupling the dashboard to chunk internals.

## GIL release feedback

The dashboard timeline continues to show GIL-release percentage and native/Python transition pressure. v2.6 additionally routes UTF-8 chunk boundary scanning through the optional native extension when available, allowing chunk scanning to release the GIL before Python performs manifest and entry commits.

## Professional dashboard assets

The v2.6 dashboard uses custom SVG iconography in the blue, purple, and orange visual system. ASCII and emoji status marks are intentionally avoided in the browser console.
